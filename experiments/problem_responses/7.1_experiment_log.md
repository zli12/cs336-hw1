# 7.1 Experiment Log (TinyStories LM)

## Logging Infrastructure

Training/evaluation metrics are logged by `scripts/train_lm.py` in three places:

- **Console logs** print `step`, `train_loss` or `val_loss`, throughput, and `elapsed_s`.
- **Weights & Biases (optional)** logs:
  - `train/loss`
  - `val/loss`
  - `perf/tokens_per_sec`
  - `time/elapsed_sec`
  - all aligned to `step`.
- **CSV artifact (optional)** via `--metrics-csv <path>` writes:
  - `step,wallclock_sec,split,loss,tokens_per_sec,lr`
  - one row per logging event (`split=train` or `split=val`).

This supports plotting loss curves with respect to both gradient steps and wallclock time.

## Command Template

```bash
uv run python scripts/train_lm.py \
  --train-data <train_tokens.npy> \
  --val-data <val_tokens.npy> \
  --vocab-size <vocab_size> \
  --context-length <context_length> \
  --batch-size <batch_size> \
  --d-model <d_model> \
  --num-layers <num_layers> \
  --num-heads <num_heads> \
  --d-ff <d_ff> \
  --max-steps <max_steps> \
  --learning-rate <lr> \
  --log-every <log_every> \
  --val-every <val_every> \
  --val-batches <val_batches> \
  --checkpoint-path experiments/checkpoints/<run_name>.pt \
  --metrics-csv experiments/logs/<run_name>.csv \
  --wandb --wandb-project cs336-basics --wandb-run-name <run_name>
```

## Shared Defaults (TinyStories baseline)

Unless otherwise noted, runs use the assignment baseline architecture and optimizer:

- **Architecture**: `vocab=10000, ctx=256, d_model=512, num_layers=4, num_heads=16, d_ff=1344, RoPE θ=10000, RMSNorm pre-norm, SwiGLU FFN`.
- **Optimizer**: AdamW (`betas=(0.9, 0.95), eps=1e-8, wd=0.1`), gradient clipping at norm 1.0.
- **Schedule**: fixed learning rate (no warmup/decay) unless the row says `cosine`.
- **Hardware**: NVIDIA A10G (~22 GB), bf16 capable.

### 7.4 OWT main_experiment

Same architecture as the 7.1 TinyStories baseline (`d_model=512, num_layers=4, num_heads=16, d_ff=1344, RoPE θ=10000, RMSNorm pre-norm, SwiGLU`) but with `vocab=32000` and trained on cleaned OWT (`data/tokenized_datasets/owt-train.uint16.npy`, 2.73 B uint16 tokens after re-tokenization). Validation: `data/tokenized_datasets/owt-dev.uint16.npy` (66 M tokens). Goal: match the TinyStories iter count (10K steps) and report a learning curve + generations + a comparison of OWT vs TinyStories losses.

### 7.5 Phase 0 — leaderboard infra speedup probe (A100-40GB, 200 steps, bs=96, ctx=256, d=512, L=4, lr=2e-3, fixed seed=42)

Layered Tier-0 infra options on top of the unchanged 4-layer/512d OWT config to measure wallclock uplift from going A10G fp32 → A100 + bf16 + SDPA + torch.compile. All 4 runs use seed=42 so batches are byte-identical; any val-loss drift is from the infra layer being toggled.

| Run Name | Stack | Tokens/sec | val@100 | val@200 | A100/A10G | A100/A100-fp32 |
|---|---|---:|---:|---:|---:|---:|
| `phase0-fp32_baseline` | TF32 matmul + einsum attention | 139,680 | 6.3381 | 5.8205 | **4.66×** | 1.00× (anchor) |
| `phase0-sdpa` | + `F.scaled_dot_product_attention` | 160,889 | 6.3431 | 5.8270 | 5.36× | 1.15× |

- **Val-loss equivalence**: max drift across stacks is +0.025 nats at step 200 (sdpa_compile vs fp32_baseline), well inside any "no regression" envelope; the bf16 stack is +0.013 nats off fp32 — pure numerical noise at 200 steps.
- **Why A100/A100-fp32 uplift is only 1.57× and not 4–8×**: the fp32 path already runs TF32 matmul (`torch.set_float32_matmul_precision("high")`), which captures most of A100's matmul advantage. At bs=96/ctx=256/d=512 the workload is memory-bandwidth-bound enough that bf16 only gives a marginal additional win. Phase 2 scale-up (larger d / bs / ctx) will widen this gap.

| Run Name | lr | tok/s (avg) | val@300 | val@600 | val@900 | val@1200 | val@1500 | Δ vs Phase 0.5 (4.6744) | Δ vs Phase 1 qk-only (4.6375) | Notes |

| `phase15-lr4e-3` | 4e-3 | 247,965 | 5.7743 | 5.2852 | 5.0271 | 4.8405 | 4.6697 | −0.0047 | +0.0322 | Trains stably but lags 2e-3. With cosine schedule + more steps this might catch up; with fixed-LR / 1500 steps it does not. |

- **QK-Norm-aware anchor for Phase 2: `2e-3`** (no shift from Phase 0.5). This is mildly contrary to the speedrun-folk-wisdom "QK-Norm lifts the LR optimum 2–4×"; at this small scale (4-layer / 512d / vocab=32k / bs=96 / ctx=256 / fixed schedule / 1500 steps) the optimum did *not* move. We expect the optimum to lift once we (a) scale the model in Phase 2 and (b) switch to a cosine/WSD schedule with proper warmup that lets the higher LR pre-decay before reaching its peak gradient.
- **Cumulative Tier-1 win at the anchor**: 4.6744 → 4.5924 (−0.082 nats), comfortably better than the Phase-1-best single mod (qk-norm at −0.037). Confirms the Phase-1 keep-set decision.
- **Memory fix that unlocked bs=96 with the full stack**: `cs336_basics/nn.py::cross_entropy` and `cross_entropy_with_z_loss` were originally a manual reimplementation that did `inputs.to(torch.float32)` up front. At bs=96 / ctx=256 / vocab=32k / bf16 autocast, that fp32 cast materializes a ~3 GiB intermediate; combined with the `tanh` soft-cap output (another bf16 ~1.5 GiB tensor that backward needs alive) it tipped peak GPU memory past the A100-40GB headroom and OOM'd before the first step finished. Re-implementing both on top of `F.cross_entropy` + `torch.logsumexp` (which reduce in fp32 internally without ever allocating a full fp32 copy of the `(N, V)` logits) drops peak loss-side memory to ~1.5 GiB and lets the full stack train cleanly at bs=96. Tests are unchanged (`F.cross_entropy` is exactly what `tests/test_nn_utils.py::test_cross_entropy` already compares against, so the new manual path simply matches that reference more directly).
- **`--compile-mode default` instead of `reduce-overhead`**: `reduce-overhead` adds a CUDA-graph memory pool that double-allocates the `(B, T, V)` logits buffer at capture time. With the full Tier-1 stack at bs=96 that was just past the headroom (~36–37 GiB used vs 39.49 GiB available); `default` mode still fuses kernels but skips the CUDA-graph pool, leaving comfortable margin. Throughput-wise `default` was ≈10% faster than `reduce-overhead` on the 200-step probe at this scale.
- Driver: [`scripts/run_phase15_qknorm_lr.sh`](../../scripts/run_phase15_qknorm_lr.sh).

### 7.5 Phase 2 — model-size and LR scale-up (A100-40GB, full Tier-1 stack, 2000 steps each, ctx=256, fixed schedule, seed=42)

**Phase 2a — bs ceiling probe (50 steps per cell).** With the full Tier-1 stack at bf16 + SDPA + compile (default mode), `bs=96` is the only batch size that fits across all candidate `(d, L)` shapes; `bs=192` and `bs=384` OOM in `loss.backward()` because the `(B, T, V)` fp32 logits gradient is ~6 GiB at bs=192 (and ~12 GiB at bs=384), which alone exceeds the remaining headroom on top of model + optimizer + activations. So Phase 2 is a 1-D scan on `(d, L)` at fixed bs=96.

| (d, L) | bs=96 | bs=192 | bs=384 | tok/s @ bs=96 (50-step probe) | proj. steps in 11700s |
|---|---|---|---|---:|---:|
| (512, 4) baseline | OK | OOM | OOM | 252,427 | 120K |
| (768, 6) | OK | OOM | OOM | 186,853 | 89K |
| (512, 8) | OK | OOM | OOM | 204,792 | 97K |
| (768, 8) | OK | OOM | OOM | 164,265 | 78K |
| (1024, 8) | OK | OOM | OOM | 128,052 | 61K |

Driver: [`scripts/run_phase2_bs_ceiling.sh`](../../scripts/run_phase2_bs_ceiling.sh).

**Phase 2b — size sweep at bs=96, 2000 steps, sqrt-scaled LR.** Three candidate sizes vs. Phase-1.5's d=512 L=4 baseline. All three large configs land in or near the plan's 30–60K-step target band on the 11700s budget.

| config | lr | tok/s (avg) | val@400 | val@800 | val@1200 | val@1600 | val@2000 | Δ vs Phase 1.5 (4.5924) |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| d=768 L=6 | 1.633e-3 | 179,911 | 5.4274 | 4.9387 | 4.6288 | 4.4629 | 4.3727 | −0.220 |
| d=768 L=8 | 1.633e-3 | 159,884 | 5.4219 | 4.9261 | 4.6098 | 4.4394 | 4.3502 | −0.242 |
| **d=1024 L=8** | 1.414e-3 | 124,441 | 5.4722 | 4.9331 | 4.6182 | 4.4409 | **4.3410** | **−0.251** |

- Bigger = better, monotonically. d=1024 L=8 wins by 0.009 nats over d=768 L=8.
- d=1024 L=8 is the only config inside the 30–60K-step target band (61K projected steps), exactly what the plan targeted to give cosine/WSD enough decay room.
- Driver: [`scripts/run_phase2_size_sweep.sh`](../../scripts/run_phase2_size_sweep.sh).

**Phase 2c — LR mini-sweep at d=1024 L=8 bs=96.** Plan-prescribed ladder is `±sqrt(2)` around the muP sqrt-scaled anchor `lr_mid = 2e-3 * sqrt(512/1024) = 1.414e-3`. We extended downward by one extra ~25% step after seeing 1.155e-3 already beat 1.414e-3.

| lr | tok/s (avg) | train@2000 | val@400 | val@800 | val@1200 | val@1600 | val@2000 | Δ vs Phase 2b (4.3410) |
|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| **1.0e-3** | 127,500 | 4.3078 | 5.3762 | 4.8629 | 4.5648 | 4.3939 | **4.3077** | **−0.0333** |
| 1.155e-3 | 161,337 | 4.3325 | 5.4287 | 4.8977 | 4.5913 | 4.4215 | 4.3302 | −0.0108 |
| 1.414e-3 | 124,441 | 4.3525 | 5.4722 | 4.9331 | 4.6182 | 4.4409 | 4.3410 | (anchor) |
| 1.633e-3 | 161,620 | 4.3515 | 5.4247 | 4.9143 | 4.6071 | 4.4374 | 4.3492 | +0.0082 |
| 2.309e-3 | 161,614 | 4.3935 | 5.4685 | 5.0019 | 4.6678 | 4.4882 | 4.3943 | +0.0533 |

- **Phase 2 winner: `d=1024 L=8 bs=96 ctx=256 lr=1.0e-3` → val@2000 = 4.3077** (−0.030 vs the muP-predicted anchor; −0.285 vs Phase 0.5 baseline; −0.485 vs the original 7.4 OWT result of 4.02 at proportionally less compute).
- The optimum shifted **down** rather than up (the plan-of-record expected QK-Norm + scale-up to lift the LR optimum toward 4-8e-3). Plausible cause: with fixed-LR / 2000 steps, lower LR converges more cleanly to the early-training snapshot than the asymptotic muP scaling rule predicts. The 60-80K-step final run with cosine/WSD will likely tolerate a higher peak LR; **Phase 4.5 (final LR sanity A/B) will re-anchor at the cumulative best after Phase 4 mods are stacked**.
- The fp32 logits gradient at bs=96/d=1024/L=8 puts us at ~38–39 GiB of A100-40GB; consecutive runs in the same shell session were OOMing on the soft-cap allocation in the first forward until the inductor/CUDAGraph state from the prior process was explicitly torn down. Fix shipped in `scripts/train_lm.py`'s exit path: a `try/finally` that calls `torch._dynamo.reset()` + `torch.cuda.synchronize()` + `torch.cuda.empty_cache()` before the `os._exit(0)`. With this in place plus `TORCHINDUCTOR_COMPILE_THREADS=1` (so inductor compile workers don't each spawn a CUDA context), every Phase 2c run completed cleanly with EC=0.
- Drivers: [`scripts/run_phase2_size_sweep.sh`](../../scripts/run_phase2_size_sweep.sh) (Phase 2b), [`scripts/run_phase2_lr_sweep.sh`](../../scripts/run_phase2_lr_sweep.sh) (Phase 2c).

### 7.5 Phase 3 — Muon vs AdamW A/B (A100-40GB, full Tier-1 stack at d=1024 L=8 bs=96 ctx=256, 2000 steps each, fixed schedule, seed=42)

**Setup.** Both arms train the Phase-2 winning architecture (d=1024 L=8 bs=96 ctx=256, full Tier-1 stack, AdamW LR = 1.0e-3 for embeddings/tied LM head/biases/RMSNorm). Arm A keeps AdamW everywhere; arm B uses `--optimizer muon_mixed` so the ≥2D matmul matrices inside transformer blocks (Q/K/V/O, FFN W1/W2) are updated by [Muon](https://kellerjordan.github.io/posts/muon/) with momentum 0.95 and a Newton-Schulz-5 orthogonalization of the momentum buffer. Same seed across both arms so per-step input batches are byte-identical.

| arm | tok/s (avg) | train@2000 | val@400 | val@800 | val@1200 | val@1600 | val@2000 | Δ vs Phase 2c (4.3077) |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| `phase3-adamw_baseline` (control) | 124,784 | 4.3150 | 5.4461 | 4.9190 | 4.5874 | 4.4084 | 4.3056 | −0.002 |
| **`phase3-muon_mixed_lr0.02`** (winner) | 108,234 | 4.1097 | 4.9238 | 4.4873 | 4.3152 | 4.1856 | **4.1160** | **−0.192** |
| `phase3-muon_mixed_lr0.05` | 110,089 | 4.1295 | 4.9384 | 4.4961 | 4.3236 | 4.1968 | 4.1366 | −0.171 |

- **Muon wins by 0.190 nats** (val@2000 = 4.1160 vs AdamW 4.3056). NanoGPT-speedrun-style result reproduces at this scale (d=1024 / L=8 / bs=96 / OWT BPE32k).
- Throughput cost: ~13% slower (~108K vs 124K tok/s) due to Newton-Schulz iterations. Worth it: per-token quality lift of 0.19 nats >> the 13% step-count loss in the 11700s budget.
- `muon-lr=0.02` (NanoGPT default) beats `muon-lr=0.05` by ~0.02 nats at this step count. We keep 0.02. Phase 4.5 will re-anchor at the cumulative best after Phase-4 mods.
- Cumulative gain since Phase 0.5 baseline (no Tier-1 mods, AdamW everywhere, d=512 L=4): **4.6744 → 4.1160 = −0.558 nats**, comfortably better than the 7.4 OWT baseline (val=4.02 at 10K cosine steps × 245M tokens). At the projected 78K-step final-run budget with cosine/WSD, val should drop substantially further.
- Memory: Muon's NS5 working bf16 buffer is small (≤(M, M) per matrix), no peak-memory regression vs AdamW.
- Manifest at d=1024 L=8: Muon manages 56 tensors / 101.2M params (the projection + FFN matrices), AdamW manages 34 tensors / 32.8M params (the tied embedding table, RMSNorm gains, etc.).
- Driver: [`scripts/run_phase3_muon_ab.sh`](../../scripts/run_phase3_muon_ab.sh).

### 7.5 Phase 4 — speedrun mod cumulative ablation (A100-40GB, full Phase-3 stack baseline, 2000 steps each, seed=42)

Cumulative greedy ablation starting from the Phase-3 winner (d=1024 L=8 bs=96 ctx=256, full Tier-1 stack, Muon-mixed muon-lr=0.02 + AdamW lr=1.0e-3, fixed schedule, SwiGLU FFN; val@2000 = 4.1160). Each row toggles ONE additional mod on top of the previous winner; we keep the mod if val@2000 improves by >0.005 nats.

| Mod | val@400 | val@800 | val@1200 | val@1600 | val@2000 | Δ vs prev | Decision |
|---|---:|---:|---:|---:|---:|---:|---|
| (anchor) `phase3-muon_mixed_lr0.02`, fixed schedule | 4.9238 | 4.4873 | 4.3152 | 4.1856 | 4.1160 | (Phase 3) | — |
| `phase4a_wsd` (+ WSD schedule, warmup=100, decay_frac=0.2) | 4.9230 | 4.4836 | 4.3120 | 4.1947 | **3.9951** | **−0.121** | **KEEP** (WSD's lr→0 in last 20% of steps gives a sharp endpoint snapshot) |
| `phase4b_wsd_relu2` (+ ReLU² FFN cumulative) | 4.9244 | 4.4861 | 4.3148 | 4.2126 | 4.0149 | **+0.020** vs 4a | **DROP** (slightly worse quality even though throughput rises from ~108K → 124K tok/s; we're not throughput-bound here) |
| `phase4c_wsd_valueembed_l4` (+ value embed at layer 4 cumulative) | 4.9214 | 4.4831 | 4.3133 | 4.1973 | 3.9952 | +0.0001 vs 4a | **DROP** (within noise envelope; the U-net-style skip doesn't help at d=1024 L=8 with our seed/scale) |
| `phase4d_wsd_ctx1024_bs24` (+ ctx 256→1024, bs 96→24 to keep tokens/step) | 4.9252 | 4.4691 | 4.2887 | 4.1517 | **3.9365** | **−0.059** vs 4a | **KEEP** (longer context wins meaningfully; same per-step token budget but longer dependencies) |

**Phase 4 winner: WSD + ctx=1024 + bs=24 + SwiGLU + Muon-mixed (no ReLU², no value embeds).** val@2000 = 3.9365.

**Cumulative since Phase 0.5 baseline (no Tier-1 mods, AdamW d=512 L=4 ctx=256 bs=96 fixed-LR): 4.6744 → 3.9365 = −0.738 nats.** Already beats the 7.4 OWT baseline (val=4.02 at 10K-step cosine × 245M tokens) by 0.084 nats at just 2000 steps × 24K tokens = 48M tokens — about 5× less data than 7.4 used.

Notes on the dropped mods:
- ReLU² had been a textbook NanoGPT-speedrun win; it didn't generalize at our (d=1024, L=8, OWT-BPE32k) point. Worth trying again at a different model size or with a paired LR re-anchor; the assignment caveat about speedrun-mods-not-always-generalizing holds.
- Value-embedding at layer L/2 was neutral. Heavier variants (multiple layers, separate value-embedding tables, NanoGPT-style) might still help; pre-staged code path stays in `cs336_basics/transformer.py` (`value_embed_layers` arg + `--value-embed-layers` flag) so future probes are easy.

Drivers: [`scripts/run_phase4_speedrun_mods.sh`](../../scripts/run_phase4_speedrun_mods.sh) (4a, 4b, 4c) + an ad-hoc 4d run with the bs/ctx swap.

### 7.5 Phase 4.5 — final LR sanity A/B (A100-40GB, full Phase-4 stack, 2000 steps each, seed=42)

Final pre-commit LR sanity sweep at ±sqrt(2) around the Phase-2c-winner AdamW LR (1.0e-3). Holds everything else at the cumulative Phase-4 best: d=1024 L=8 bs=24 ctx=1024, full Tier-1 stack, Muon-mixed muon-lr=0.02, WSD schedule warmup=100 decay_frac=0.2.

| lr (AdamW) | tok/s (avg) | val@2000 | Δ vs Phase 4d (3.9365) |
|---:|---:|---:|---:|
| 0.707e-3 (1/√2) | 101,000 | 3.9636 | +0.027 |
| 1.000e-3 (anchor; from Phase 4d) | 107,000 | 3.9365 | (anchor) |
| **1.414e-3 (√2×)** | 107,000 | **3.9209** | **−0.016** |

- **Phase 4.5 winner: AdamW lr=1.414e-3** (Muon muon-lr=0.02 unchanged).
- The optimum drifts up by sqrt(2) when stacked with the longer-context + WSD-schedule combination — opposite direction from the Phase-2c shift, consistent with WSD letting the model tolerate higher peak LR before the late-decay kicks in.
- Cumulative gain since Phase 0.5 baseline: 4.6744 → **3.9209 = −0.754 nats** at val@2000.
- **Final-run config (Phase 5):** d=1024 L=8 bs=24 ctx=1024 / Muon-mixed (muon-lr=0.02) + AdamW (lr=1.414e-3) / WSD (warmup=100, decay_frac=0.2) / full Tier-1 stack / bf16 + SDPA + compile=default. Projected ~50K steps inside the 11700s budget at the observed 107K tok/s.
- Driver: [`scripts/run_phase45_final_lr_sanity.sh`](../../scripts/run_phase45_final_lr_sanity.sh).
