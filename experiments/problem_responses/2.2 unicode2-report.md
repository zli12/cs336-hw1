# 2.2 Problem (unicode2): Unicode Encodings

## Problem Statement

**Problem (unicode2): Unicode Encodings (3 points)**  
(a) What are some reasons to prefer training our tokenizer on UTF-8 encoded bytes, rather than UTF-16 or UTF-32? It may be helpful to compare the output of these encodings for various input strings.  
Deliverable: A one-to-two sentence response.  
  
(b) Consider the following (incorrect) function, which is intended to decode a UTF-8 byte string into a Unicode string. Why is this function incorrect? Provide an example of an input byte string that yields incorrect results.  
`def decode_utf8_bytes_to_str_wrong(bytestring: bytes):`  
`    return "".join([bytes([b]).decode("utf-8") for b in bytestring])`  
`>>> decode_utf8_bytes_to_str_wrong("hello".encode("utf-8"))`  
`'hello'`  
Deliverable: An example input byte string for which `decode_utf8_bytes_to_str_wrong` produces incorrect output, with a one-sentence explanation of why the function is incorrect.  
  
(c) Give a two byte sequence that does not decode to any Unicode character(s).  
Deliverable: An example, with a one-sentence explanation.

## Answers

## (a)
**Question:** Why prefer UTF-8 bytes over UTF-16/UTF-32 for tokenizer training?  
**Answer:** UTF-8 is compact for common web text (especially ASCII-heavy English), universally used in modern text pipelines, and naturally byte-oriented, which makes byte-level tokenization simple and portable; UTF-16/UTF-32 use wider code units, add endianness/surrogate complications, and usually increase storage without improving tokenizer behavior.

## (b)
**Question:** Why is `decode_utf8_bytes_to_str_wrong` incorrect, and give an example input?  
**Answer:** Example input `b"\xc3\xa9"` (UTF-8 for `"é"`) fails because the function decodes each byte independently even though UTF-8 characters can span multiple bytes, so it breaks multibyte sequences and can raise decoding errors or produce wrong results.

## (c)
**Question:** Give a two-byte sequence that does not decode to any Unicode character(s).  
**Answer:** `b"\xff\xff"` is invalid in UTF-8 because `0xFF` is not allowed as a valid UTF-8 start or continuation byte, so the sequence cannot decode to any Unicode character.
