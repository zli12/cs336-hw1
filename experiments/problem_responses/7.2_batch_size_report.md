# 7.2 Batch-Size Sweep Report (`batch_size_experiment`)

## Setup

- **Hardware**: NVIDIA A10G (~22 GB VRAM), bf16 capable.
- **Architecture**: TinyStories baseline (`vocab=10000, ctx=256, d_model=512, num_layers=4, num_heads=16, d_ff=1344, RoPE θ=10000, pre-norm RMSNorm, SwiGLU`).
- **Optimizer**: AdamW (`betas=(0.9, 0.95), eps=1e-8, wd=0.1`), gradient clipping at norm 1.0.
- **Schedule**: fixed learning rate, no warmup or decay (matches the 7.2 LR sweep methodology).
- **Compute budget per run**: `max_steps = 3000`. Validation runs every 200 steps over 20 batches.
- **Why fix steps not tokens?** The deliverable is a learning-curve comparison; fixing optimizer steps keeps the *update count* identical, isolating how much each batch buys per update. Fixing tokens would have collapsed the bs=1 run to a single second of wallclock and made the bs sweep degenerate. We separately report wallclock so the reader can see the efficiency story.

## Learning-rate scaling rule

Smaller batches have noisier gradients and (in the SGD-noise-scale view) prefer smaller learning rates; larger batches are well-conditioned and tolerate larger LRs. We picked `lr(bs) = 2e-3 * sqrt(bs / 128)`, anchored to the LR-tuned `bs=128, lr=2e-3` from the 7.2 LR sweep.

| bs | sqrt-scaled lr | rounded |
|---:|---:|---:|
| 1   | 1.77e-4 | 2e-4   |
| 8   | 5.0e-4  | 5e-4   |
| 32  | 1.0e-3  | 1e-3   |
| 64  | 1.41e-3 | 1.4e-3 |
| 128 | 2.0e-3  | 2e-3 (anchor) |
| 192 | 2.45e-3 | 2.5e-3 |

To validate the scaling rule we also ran two **controls** at the anchor `lr=2e-3`: `bs=8, lr=2e-3` and `bs=192, lr=2e-3`.

## Findings

**1. At equal optimizer steps, larger batch ⇒ lower loss, monotonically.** Each step processes more tokens, so the model sees more data and gradient estimates are less noisy. From `bs=1` (val 3.08) to `bs=192` (val 1.52), the gap is dramatic. Returns diminish though: doubling from `bs=64 → bs=128` cut val loss by 0.13, while `bs=128 → bs=192` (a 1.5x jump) cut it only 0.07. This is the classic critical-batch-size pattern: above some scale, additional batch does not buy proportional generalization improvement.

**2. At equal wallclock seconds, smaller-but-fast-enough batches dominate early, large batches eventually catch up.** From the wallclock panel of `7.2_batch_size_sweep.svg`, `bs=32` reaches val ~2.0 around 100s while `bs=128` is still at val ~3.0; but past ~600s the larger batch overtakes because the smaller batch has exhausted its low-LR convergence. There is no single best batch size; the tradeoff depends on the compute budget.

**3. GPU utilization plateaus by `bs≈32`.** Steady-state `tok/s` reaches ~52k at `bs=8`, ~61k at `bs=32`, and saturates ~63k at `bs=64`/`bs=128`. `bs=192` actually drops to ~56k, indicating diminishing returns and likely some allocator/cache pressure. `bs=1` is dramatically slower per token (~10k tok/s) because Python/CUDA-launch overhead dominates. **Bottom line: matrix multiplies are tensor-core efficient once you fill the GPU, but going from `bs=64` to `bs=192` does not raise throughput, only memory.**

**4. The sqrt-LR rule is helpful at small batches and neutral at large batches.**
   - `bs=8, lr=2e-3` (anchor) gave **val 2.487** vs `bs=8, lr=5e-4` (sqrt) at **val 2.303**. With only 256 tokens per gradient step, gradient noise is high; the anchor LR is too aggressive and the loss visibly oscillates around step 1500-3000 in the controls plot.
   - `bs=192, lr=2.5e-3` (sqrt) gave **val 1.520** vs `bs=192, lr=2e-3` (anchor) at **val 1.523** - effectively identical. The anchor LR is below the well-conditioned regime's optimum, but not catastrophically so. (At very large batch sizes, the linear-LR rule could be a slight win, but a 1.5x scale-up from 128 is too small to see it.)

**5. GPU memory limit on the A10G.** With `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`, `bs=192` was the largest batch that ran without OOM. Earlier failures at `bs=64` were due to the GPU being concurrently occupied by another training process; once the GPU was free, `bs=64` ran cleanly. We did not attempt `bs=256`; it would likely OOM given that activations scale linearly with batch and `bs=192` already consumes most of the 22 GB.

## Practical takeaway

For this 4-layer / `d_model=512` model on an A10G:
- `bs=128 → 192` is the sweet spot on a single A10G: highest throughput is at `bs=64-128`, but lowest val loss in 3000 steps is at `bs=192`.
- Below `bs=32`, you waste the GPU (Python overhead, undertensor-cored kernels).
- If you are LR-sensitive (distributed training, switching hardware), keep an eye on the LR; sqrt-scaling holds well over a 24x batch range here, but small batches need it more than large batches.
- Use the anchor `lr=2e-3` only if you stay near `bs=128`. At very small batches, drop the LR; at very large batches you can keep it without much loss.

## Reproduction

Single sweep driver: `/tmp/run_bs_sweep.sh` (a serial bash loop running each `(bs, lr)` for 3000 steps). Logs in `experiments/logs/bs-sweep-bs<BS>-fixed-lr<LR>.csv` and `.console.log`. Plots reproduced via:

```bash
uv run python scripts/plot_runs.py --kind batch-sweep \
  --inputs <CSVs in increasing batch-size order> \
  --labels "bs=1, lr=2e-4" ... "bs=192, lr=2.5e-3" \
  --out experiments/problem_responses/7.2_batch_size_sweep.svg \
  --title "Batch-size sweep (fixed LR, sqrt-scaled, 3000 steps)"
```
