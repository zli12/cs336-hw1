# 2.5 BPE Training Report

## TinyStories (2.5)

### (a) Train tokenizer + report runtime/memory/longest token

- **Config:** `vocab_size=10000`, special token `<|endoftext|>`.
- **Artifacts:** `experiments/tinystories-bpe-10k-vocab.json`, `experiments/tinystories-bpe-10k-merges.txt`.
- **Runtime:** `62.644 s` (`0.017401 h`).
- **Memory:** `1.275 GiB` USS (`1.711 GiB` RSS sum).
- **Longest token:** `15` bytes, `b' accomplishment'`.
- **Interpretation:** This is reasonable because BPE on English text often learns frequent leading-space wordpieces.

### (b) Profile summary

- **Main bottlenecks:** multiprocessing coordination/waiting (`concurrent.futures`/poll/wait) and merge-heap operations (`max_pair_heap.pop_max`, heap push/pop/update).
- **Takeaway:** IPC/process coordination and heap maintenance dominate runtime more than pure regex pretokenization.

## OpenWebText (treated as follow-up subproblem)

### (a) Train tokenizer + report runtime/memory/longest token

- **Config:** `vocab_size=32000`, special token `<|endoftext|>`.
- **Artifacts:** `experiments/owt-bpe-32k-vocab.json`, `experiments/owt-bpe-32k-merges.txt`.
- **Runtime:** `8736.027 s` (`2.426674 h`).
- **Memory:** `25.646 GiB` USS (`25.648 GiB` RSS sum).
- **Longest token:** `64` bytes, repeated mojibake-like bytes.
- **Interpretation:** This is plausible for noisy web corpora with encoding artifacts.

### (b) TinyStories vs OpenWebText tokenizer comparison

- **OpenWebText tokenizer:** learns longer/noisier web-specific patterns (including encoding artifacts).
- **TinyStories tokenizer:** learns cleaner, narrative English wordpieces (e.g., `b' accomplishment'`).
- **Reason:** OpenWebText has broader, noisier distribution than TinyStories, so BPE merges reflect that diversity.
