# 2.7 Tokenizer Experiments Summary

## Key Metrics
- **Sample size:** 10 documents per dataset.
- **TinyStories tokenizer on TinyStories:** `4.0908` bytes/token.
- **OpenWebText tokenizer on OpenWebText:** `4.3011` bytes/token.
- **TinyStories tokenizer on OpenWebText:** `3.0281` bytes/token.
- **Throughput:** `3,364,009` bytes/sec (~`3.36 MB/s`).
- **Estimated time for 825 GiB:** `73.15` hours.

## Deliverable Answers

### (a) Compression ratio
On 10 sampled documents, the TinyStories tokenizer achieves **4.09 bytes/token** on TinyStories and the OpenWebText tokenizer achieves **4.30 bytes/token** on OpenWebText. Both are reasonable for byte-level BPE, with OWT reflecting a broader/noisier text distribution.

### (b) Tokenizing OWT with the TinyStories tokenizer
When the OpenWebText sample is tokenized with the TinyStories tokenizer, compression drops to **3.03 bytes/token** (vs **4.30** with the OWT tokenizer), showing substantially poorer domain fit. In practice this means more token fragmentation on web text.

### (c) Throughput + Pile estimate
Measured tokenizer throughput is about **3.36 MB/s** (~3,364,009 bytes/s). At this speed, tokenizing **825 GiB** would take roughly **73.15 hours**.

### (d) Encoded datasets and `uint16` choice
TinyStories and OpenWebText train/dev datasets were encoded into NumPy token-ID arrays for later language-model training. `uint16` is appropriate because token IDs stay below 65,536 (`max_id=9999` for TinyStories and `31999` for OWT), so it preserves values while reducing storage.
