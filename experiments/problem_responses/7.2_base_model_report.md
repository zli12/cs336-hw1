# 7.2 TinyStories Baseline Report (A10G)

## Run Goal

Train a single baseline TransformerLM on TinyStories with the assignment architecture and full token budget:

- `vocab_size=10000`
- `context_length=256`
- `d_model=512`
- `d_ff=1344`
- `num_layers=4`
- `num_heads=16`
- `rope_theta=10000`

## Hardware

- GPU: `NVIDIA A10G` (`~23 GB` VRAM)

## Data

- Train tokens: `data/tokenized_datasets/tinystories-train.uint16.npy`
- Validation tokens: `data/tokenized_datasets/tinystories-dev.uint16.npy`

## Baseline Training Command

```bash
uv run python scripts/train_lm.py \
  --train-data data/tokenized_datasets/tinystories-train.uint16.npy \
  --val-data data/tokenized_datasets/tinystories-dev.uint16.npy \
  --vocab-size 10000 \
  --context-length 256 \
  --batch-size 128 \
  --d-model 512 \
  --num-layers 4 \
  --num-heads 16 \
  --d-ff 1344 \
  --rope-theta 10000 \
  --max-steps 10000 \
  --learning-rate 0.002 \
  --weight-decay 0.1 \
  --beta1 0.9 \
  --beta2 0.95 \
  --eps 1e-8 \
  --max-grad-norm 1.0 \
  --device cuda \
  --log-every 20 \
  --val-every 200 \
  --val-batches 20 \
  --checkpoint-path experiments/checkpoints/tinystories-base-a10g.pt \
  --checkpoint-every 200 \
  --metrics-csv experiments/logs/tinystories-base-a10g.csv
```

## Token Budget Check

- `batch_size * context_length * max_steps`
- `128 * 256 * 10000 = 327,680,000` tokens (matches target full baseline budget)

## Artifacts

- Metrics CSV: `experiments/logs/tinystories-base-a10g.csv`
- Console log: `experiments/logs/tinystories-base-a10g.console.log`
- Final checkpoint: `experiments/checkpoints/tinystories-base-a10g.pt`

## Throughput Notes

- Steady-state train throughput was about `64.3k tokens/s` on most train log points.
- Lower values around `46.2k tokens/s` appear right after validation checkpoints (expected cadence overhead).
- The very first logged step has an inflated throughput estimate due to startup timing and should be ignored for steady-state interpretation.

## Interpretation

This full-token baseline converged smoothly on A10G and produced stable late-training validation behavior in the `1.45-1.47` loss range. The best validation point happened before the final step (`8600`), suggesting follow-up runs should tune optimization schedule (learning rate/warmup/decay) to improve end-of-run validation performance.
