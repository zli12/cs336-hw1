# 7.1 Experiment Log (TinyStories LM)

## Logging Infrastructure

Training/evaluation metrics are logged by `scripts/train_lm.py` in three places:

- **Console logs** print `step`, `train_loss` or `val_loss`, throughput, and `elapsed_s`.
- **Weights & Biases (optional)** logs:
  - `train/loss`
  - `val/loss`
  - `perf/tokens_per_sec`
  - `time/elapsed_sec`
  - all aligned to `step`.
- **CSV artifact (optional)** via `--metrics-csv <path>` writes:
  - `step,wallclock_sec,split,loss,tokens_per_sec,lr`
  - one row per logging event (`split=train` or `split=val`).

This supports plotting loss curves with respect to both gradient steps and wallclock time.

## Command Template

```bash
uv run python scripts/train_lm.py \
  --train-data <train_tokens.npy> \
  --val-data <val_tokens.npy> \
  --vocab-size <vocab_size> \
  --context-length <context_length> \
  --batch-size <batch_size> \
  --d-model <d_model> \
  --num-layers <num_layers> \
  --num-heads <num_heads> \
  --d-ff <d_ff> \
  --max-steps <max_steps> \
  --learning-rate <lr> \
  --log-every <log_every> \
  --val-every <val_every> \
  --val-batches <val_batches> \
  --checkpoint-path experiments/checkpoints/<run_name>.pt \
  --metrics-csv experiments/logs/<run_name>.csv \
  --wandb --wandb-project cs336-basics --wandb-run-name <run_name>
```

## Shared Defaults (TinyStories baseline)

Unless otherwise noted, runs use the assignment baseline architecture and optimizer:

- **Architecture**: `vocab=10000, ctx=256, d_model=512, num_layers=4, num_heads=16, d_ff=1344, RoPE θ=10000, RMSNorm pre-norm, SwiGLU FFN`.
- **Optimizer**: AdamW (`betas=(0.9, 0.95), eps=1e-8, wd=0.1`), gradient clipping at norm 1.0.
- **Schedule**: fixed learning rate (no warmup/decay) unless the row says `cosine`.
- **Hardware**: NVIDIA A10G (~22 GB), bf16 capable.

### 7.4 OWT main_experiment

Same architecture as the 7.1 TinyStories baseline (`d_model=512, num_layers=4, num_heads=16, d_ff=1344, RoPE θ=10000, RMSNorm pre-norm, SwiGLU`) but with `vocab=32000` and trained on cleaned OWT (`data/tokenized_datasets/owt-train.uint16.npy`, 2.73 B uint16 tokens after re-tokenization). Validation: `data/tokenized_datasets/owt-dev.uint16.npy` (66 M tokens). Goal: match the TinyStories iter count (10K steps) and report a learning curve + generations + a comparison of OWT vs TinyStories losses.

#### 7.4 LR probe (bs=64, fixed schedule, 3000 steps each)

Re-tuning LR for OWT because the spec note says batch / LR may need to change (and bs=128 OOMs on A10G with the 32k-vocab logits — see Phase 1 notes below).

| Run Name | bs | lr | Best Val Loss | Step @ Best | Wall @ End (s) | Notes |
|---|---:|---:|---:|---:|---:|---|
| `owt-lr-probe-fixed-lr1e-3` | 64 | 1e-3 | 4.626 | 3000 | 1188 | Loss still dropping at step 3000 (−0.036 nats over last 200 steps); undertrained. |
| `owt-lr-probe-fixed-lr2e-3` | 64 | 2e-3 | **4.600** | 3000 | 1188 | **LR-probe winner.** Same downward slope at 3000. |
| `owt-lr-probe-fixed-lr3e-3` | 64 | 3e-3 | 4.624 | 3000 | 1188 | Essentially tied with 1e-3; LR optimum is shallow around 2e-3. |

#### 7.4 Batch-size probe (3000 steps, sqrt-LR scaled from `bs=64, lr=2e-3`)

A10G has 22 GB VRAM. With `vocab=32000`, the logits tensor (and its grad) at training time is `bs × ctx × vocab × 4 bytes` ≈ 4 GiB at bs=128, vs ≈ 1.3 GiB on TinyStories at the same bs — so the same batch sizes that fit on TinyStories no longer fit on OWT.

| Run Name | bs | lr | Best Val Loss | Step @ Best | Wall @ End (s) | Notes |
|---|---:|---:|---:|---:|---:|---|
| `owt-bs-sweep-bs128-fixed-lr2.83e-3` | 128 | 2.83e-3 | — | — | — | **OOM** at step 1 with `expandable_segments=True`: GPU at 20.3 / 22 GiB before the logits-grad allocation. |
| `owt-bs-sweep-bs96-fixed-lr2.45e-3` | 96 | 2.45e-3 | **4.458** | 3000 | 2417 | Fits at ~20.0 GiB. Per-step time ~2.0× bs=64 for 1.5× tokens. bs=96 beats bs=64 anchor by **−0.142 nats** at step 3000 (4.458 vs 4.600). Used for the full-budget run below. |

**Bs choice:** bs=96 wins on val/step at constant iteration count (the assignment-relevant metric), at the cost of 2× wallclock per step. bs=64 wallclock-efficiency is actually higher (reaches val 4.6 in ~1200 s vs bs=96's ~2400 s for the same val), but since the deliverable specifies "same total training iterations as TinyStories" (10K steps), we optimize for val@10K-steps and use bs=96.

#### 7.4 Full-budget run (cosine schedule, 10K steps; matches TinyStories iter count)

Cosine warmup=500, `lr_min=lr_max/10`, decay over the full 10K cycle (TinyStories `7.2` showed cosine wins +0.10 nats at the same compute budget; the win was decay-driven). On OWT the decay phase was even more impactful: val dropped from `4.458` at step 3000 (end of bs=96 fixed-LR probe) to `4.0167` at step 10000 — **−0.441 nats** over the 7K-step decay tail.

Val still trending down at step 10000 (last 1K steps drop ~-0.03 nats), edge of further improvement under the 10K-step budget. Checkpoint: `experiments/checkpoints/owt-final-cosine-bs96.pt`. |

### 7.5 Leaderboard

Same architecture/data as 7.4 OWT, but with three groups of modifications stacked.
Phase A: pure compute speedups (bf16 autocast + torch.compile + torch.nn.functional
SDPA / FlashAttention). Phase B: architecture/loss changes A/B-tested against the
Phase-A baseline, one at a time at fixed `lr=2.45e-3`, `bs=96`, 3K steps. Phase C:
the final 1.5h-on-A10G run with all winners stacked + cosine schedule.

#### 7.5 Phase A: compute speedup (1K-step OWT, fixed `lr=2.45e-3`, `bs=96`)

Same model + data as 7.4 OWT baseline; only the kernel/dtype changed. Goal: verify
the wallclock win and confirm matched per-step quality.

| Run Name | dtype | compile | attn-kernel | Best Val Loss | Step @ Best | Tokens/s | Wall @ 1K (s) | Notes |
|---|---|---|---|---:|---:|---:|---:|---|
| 7.4 baseline @ step 1000 (cosine) | fp32 | off | einsum | 4.957 | 1000 | 30,905 | 785 | Reference reading from the 7.4 OWT cosine run at the same step. |
| `owt-phaseA-probe-bf16-compile-sdpa-bs96-lr2.45e-3` | bf16 | on | torch | **4.891** | 1000 | 84,637 | 347 | Phase-A probe (fixed-LR for cleanest A/B). **2.26× wallclock speedup, −0.066 nats** at step 1000. The −0.066 nats is the no-warmup advantage at fixed LR (Phase A starts at full LR from step 1; 7.4 baseline was ramping); per-step quality is matched. |

Tokens/s improvement of `~2.7×` (84,637/30,905) is mostly from bf16 + Flash SDPA
(matmul-bound + attention-bound layers each get ~2× from bf16 and ~1.3× from the
fused causal kernel); torch.compile fuses pointwise ops on top of that. The
bf16 path is numerically equivalent to fp32 here because cross_entropy and
RMSNorm both upcast to fp32 internally.

#### 7.5 Phase B: architecture / loss A/B (3000 steps each, on top of Phase-A speedups)

All comparisons against `owt-phaseB-probe-baseline-noB` (same Phase-A speedups,
nothing else). Fixed schedule, same seed, same LR; only the named flag differs.
Δ-nats are `mod − baseline` at step 3000 (negative = mod helps).

| Run Name | Mod | Cmd flag | Best Val Loss | Δ-nats @3K | Tokens/s | Wall @ 3K (s) | Verdict |
|---|---|---|---:|---:|---:|---:|---|
| `owt-phaseB-probe-baseline-noB` | (none, Phase-A only) | — | 4.463 | 0.000 (ref) | 87,700 | 875 | Reference for B1–B4. |
| `owt-phaseB-probe-tie` | Weight tying | `--tie-embeddings` | **4.406** | **−0.057** | 88,500 | 874 | **WIN.** Embedding init scaled to `1/√d_model` so the tied LM head doesn't start with huge logits. Loss starts ~0.16 nats *behind* baseline (smaller-σ embed = smaller logits = higher entropy at init), crosses over at step ~1200, finishes 0.057 nats ahead. Trajectory still narrowing at 3K steps — gain may grow at longer horizons. Saves `vocab×d_model = 16.4M` params. |
| `owt-phaseB-probe-qknorm` | QK-norm | `--qk-norm` | **4.418** | **−0.045** | 86,500 | 905 | **WIN.** Per-head `RMSNorm(head_dim)` on Q and K *before* RoPE (Llama-3 / Qwen-2.5 style). Roughly constant 0.045-nat advantage from step 200 onward. Costs ~3% wallclock for 2 extra norms per attention block. |
| `owt-phaseB-probe-zloss` | PaLM z-loss | `--z-loss-coef 1e-4` | **4.446** | **−0.017** | 87,700 | 873 | **WIN (small).** Adds `1e-4 * mean(log_normalizer²)` to the train loss, keeping the partition-function magnitude bounded. Eval-time loss is the standard NLL only (`z_loss_coef` defaults to 0 in `cs336_basics.nn.cross_entropy`), so val numbers are directly comparable. ~0 wallclock cost. |
| `owt-phaseB-probe-pgwd` | Param-group WD | `--param-group-wd` | 4.483 | **+0.020** | 87,500 | 875 | **LOSE.** Disabling weight decay on RMSNorm gains and Embedding weights (standard "no-decay-on-no-Linear" recipe) makes things slightly *worse* at this scale + LR. Hypothesis: WD on the embedding helps regularize at small scale where the embedding is heavily reused on every step. Skipped from final stack. |

#### 7.5 Final 1.5h leaderboard run on A10G (all winners stacked + cosine schedule)

Mods enabled: bf16 autocast, torch.compile, torch SDPA, weight tying (with reduced
embed init), QK-norm, and z-loss (coef `1e-4`). `--param-group-wd` is intentionally
OFF based on the B4 result. Cosine warmup=500 over 17 K total steps (lr peaks at
2.45e-3, decays to 2.45e-4); same `bs=96, ctx=256` as 7.4. Throughput ~87 K
tokens/s, so 17 K steps is ~4900 s ≈ 82 minutes — comfortably under the 1.5 h
(5400 s) budget.

## How To Produce Curves

1. Run training with `--metrics-csv`.
2. Plot `loss` vs `step` grouped by `split` using `scripts/plot_runs.py`.
3. Plot `loss` vs `wallclock_sec` grouped by `split` for the wallclock view.
4. For each new ablation/hyperparameter run, append a row to the appropriate table above.
